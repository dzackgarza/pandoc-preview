# Reference: Pandoc-Aware Editor Syntax Highlighting (CodeMirror 6)

**When this applies:** implementing the "extremely good pandoc-aware syntax highlighting" feature ([Feature Catalogue and Implementation Status](feature-catalogue-and-implementation-status)). Scoped 2026-06-13 via research agent; unsolved in-repo so far.

**In-repo baseline (verified):** the target app and both prior iterations use only stock `@codemirror/lang-markdown` (`markdown({ base: markdownLanguage, codeLanguages })`, `src/lib/components/EditorPane.svelte:56`) — no pandoc-specific extensions. Fenced divs, citations, math, attributes, wikilinks get no special highlighting today.

**The mechanism (CM6 standard path):** extend the markdown parser via `@lezer/markdown`'s `MarkdownConfig` API, composed into `markdown({ extensions: [...] })` — one small parser per construct (`InlineParser`/`BlockParser`/`defineNodes`/`styleTags`), NOT a full custom grammar and NOT StreamLanguage decorations for the markdown layer. Paired inline delimiters use `cx.addDelimiter({resolve,mark}, ...)` (model: built-in `Strikethrough`); single tokens use `cx.addElement(cx.elt(...))` (model: `Emoji`); blocks use a `BlockParser` consuming `cx.nextLine()` or `startComposite()` for nesting. Colors: `Tag.define()` per node → `HighlightStyle.define([...])` + `syntaxHighlighting(...)`.

**Definitive reference — Zettlr** (`github.com/Zettlr/Zettlr`, `source/common/modules/markdown-editor/parser/`). CM6 + pandoc-markdown. **GPL-3.0 → design reference only, reimplement; do not copy code.** One file per construct: `math-parser.ts`, `citation-parser.ts`, `pandoc-div-span-parser.ts` (composite `startComposite()` for `::: {.theorem}`), `pandoc-attributes-parser.ts`, `footnote-parser.ts`, `frontmatter-parser.ts` (`parseMixed`→`@codemirror/lang-yaml`); tags in `util/custom-tags.ts` + `theme/syntax.ts`. Obsidian (closed, CM6) corroborates the architecture; `github.com/erykwalder/lezer-markdown-obsidian` (MIT, stale 2023) has adaptable math-delimiter + wikilink configs. PanWriter is CM5 (not applicable).

**Math sub-highlighting inside `$…$`/`$$…$$` — two options:**
- (b, recommended for the spike) Zettlr's trick: `const stex = StreamLanguage.define(stexMath)` (from `@codemirror/legacy-modes/mode/stex`), then call `stex.parser.parse(contents)` directly and splice via `ctx.elt(...)` inside your own math parser — no Lezer LaTeX grammar needed.
- (a, richer) `codemirror-lang-latex` (TeXlyre, MIT, v0.x — a real Lezer `LRParser` usable as a `parseMixed` inner parser). Young/single-maintainer; pin the version, isolate behind the math-nesting boundary so it can be swapped back to stex.

MathJax stays the *render* engine; this LaTeX highlighting is cosmetic and editor-only — that separation is correct.

**Packages:** depend on `@codemirror/lang-markdown`, `@lezer/markdown`, `@lezer/highlight`, `@lezer/common`, `@codemirror/legacy-modes` (stex), `@codemirror/lang-yaml`; optional `codemirror-lang-latex`. Custom (no package covers them): fenced divs, spans, attributes `{#id .class}`, citations `@key`/`[@key]`, the `$`/`$$` delimiters, the nesting glue. **Do NOT use:** `@codemirror/lang-stex` (does not exist — use the legacy-mode), `lezer-latex` (AGPL, abandoned), KaTeX/MathJax widget packages (rendering, not highlighting).

**Recommended:** reimplement Zettlr's architecture as a handful of `MarkdownConfig` extensions added to `EditorPane.svelte:56`. Construct priority by math-writing value: inline+display math → fenced divs (theorem envs) → citations → attributes/spans → footnotes → wikilinks → definition lists. Spike (smallest proof): add `extensions: [InlineMath, DisplayMath, PandocDiv]`, a `HighlightStyle` for the new tags, and the stex-nesting trick; open a doc with `$x^2$`, a `$$…$$` block, and `::: {.theorem} … :::` and confirm each highlights distinctly with no regression — one screenshot is the acceptance artifact.

**Open questions:** (1) **Philosophy-rule scope** — does "pandoc constructs are never parsed by custom regex code" ([Founding Philosophy: Exact Pandoc Preview](founding-philosophy-exact-pandoc-preview)) forbid editor-side highlighting parsers? Research read (high confidence): NO — that rule governs the *render pipeline* (where misparsing corrupts the document); highlighting is cosmetic, self-correcting, and must never feed the renderer. Confirm with user before building. (2) Definition lists — no reference found, hardest novel parser. (3) Raw-TeX blocks (` ```{=latex} `) likely via the `codeLanguages` path, unverified. (4) `codemirror-lang-latex` nested-in-markdown use is undocumented — validate in the spike.
