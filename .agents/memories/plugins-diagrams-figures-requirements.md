# Plugins, Diagrams, Figures Requirements

**When this applies:** designing the plugin system, diagram tool integrations, figure library, or Zotero integration.
User-stated 2026-06-13.

**The renderer itself is the pandoc plugin (user, 2026-06-13; corrected 2026-06-20).** The product is one editor -> pandoc rendering plugin -> output pane pipeline, and **plugins own their own configuration UI entirely** (clarified 2026-06-14): each plugin supplies a configuration-manager *command* the app merely launches; the app renders no config page itself and forces no config-UI schema.
The full model is in [Renderer Plugin Architecture](renderer-plugin-architecture): modes are source-type + pandoc command/input/output/view orchestration, not alternate app-owned renderers.
Everything below about the firewall, the richness bar, and individual management applies to pandoc output plugins too.
**The entire pandoc flag-configuration setup is gum-driven (in a kitty popup) inside the pandoc plugin** — invisible to the app ([Shipped Config vs Runtime Defaults](shipped-config-vs-runtime-defaults)).

**The plugin firewall (user, 2026-06-13 — architectural doctrine, not just a feature):** many "tight OS integration" requirements should NOT be app-owned features at all.
"One doesn't need the app to own Ctrl-P launching a file picker, or even navigating or managing the figures directories" — a more generalized plugin API lets these be standard scripts that use fzf/dmenu, define their own kitty popup terminals, launch external AGS GUIs, or run custom gum configuration wizards (OS-integration scripts need only modest generalization; renderer/export plugins need the fuller capability in the richness bar below).
The design task is to decide **which features modularize behind the plugin firewall "for completely independent development that drastically reduces blast radius for future changes."** Candidates named so far: quick-open/file picking, figures-directory navigation and management.
When scoping any OS-integration feature, the first question is whether it belongs behind the firewall — app-owned code is the fallback position, not the default.

**Richness bar (user, 2026-06-13):** the plugin system must be rich enough that **all export types are handled as standalone plugins, and the diagram tools as plugins — all managed individually.** Exports and diagram integrations are not app features that happen to be pluggable; they are plugins, period.
Consequences for the API floor: a plugin must be able to (a) receive the pre-populated context (file path, buffer contents, metadata) plus its own config, (b) run its owned command surface, with pandoc output plugins routing document conversion through pandoc plus filters/templates/tools ([Rendering Pipeline Requirements: Filters, MathJax, References](rendering-pipeline-requirements-filters-mathjax-references)), (c) return a structured result the app surfaces (artifact path / logs / exit status), (d) be **individually managed** — installed, enabled/disabled, updated, and QC'd per-plugin, not as a monolithic bundle, (e) **own its own config declaratively** — the standard plugin-interface pattern: ship a config schema in a standardized declarative format (JSON-Schema-class) as part of the manifest, and the app runs ONE generic validator that checks each plugin's config section against that plugin's own declared schema, so the shipped app validates every plugin config without containing any plugin-specific knowledge; **that schema is used ONLY for load-time validation (fail-loud), never to render UI** — config *editing* is owned by the plugin's own launched configuration-manager command (clarified 2026-06-14; the app renders no plugin config page — [Renderer Plugin Architecture](renderer-plugin-architecture)) ([Shipped Config vs Runtime Defaults](shipped-config-vs-runtime-defaults)). The app validates its own core schema the same generic way.
(f) **contribute its own diagnostic checks** to the doctor battery (e.g. the pandoc plugin contributes pandoc-executable / pandoc-invocation / required-filter / template-existence checks — [Doctor Contract (D1–D5)](doctor-contract)). Precedent: the original Express-era implementation already bundled HTML/LaTeX/PDF exports as plugins, so this is a restoration of proven shape, not speculation.

**This necessitates a more capable, expressive plugin system than "scripts with variables."** Because pandoc renderer/output configuration AND validation are now plugin-owned ([Renderer Plugin Architecture](renderer-plugin-architecture)), the plugin API cannot be the minimal seed shape; it must carry config-schema declaration, config-validation ownership, config-page exposure, and diagnostic contribution (items e–f above).
The simple drop-in-script case below remains the easy path for trivial plugins, but it is the floor, not the ceiling.

**Plugin system = drop-in scripts:** "a simple plugin system which more or less just allows defining scripts that have access to some pre-populated variables (e.g. the file path or buffer contents as a string or other various metadata)." Scripts are **quality-checked in isolation before being integrated**; the plugin menu is **dynamically populated from the drop-ins**. The seed framing "no plugin API surface beyond variable injection + execution + structured result" is **superseded (2026-06-13)** — that floor still describes the trivial plugin, but the system as a whole must also carry config-schema/validation/config-page ownership and diagnostic contribution (richness-bar items e–f above), because renderers are plugins and own their own config and validation.
The original implementation's working shape (Express era, mine for structure not code): **TOML manifests, category-grouped menu, spawn-based execution with variable interpolation, bundled pandoc-backed HTML/LaTeX/PDF export plugins**; metadata `{id, name, description, category}`. Its result type allowed `{ok?: boolean}` — banned now; results must be structured success/failure.

**Diagram tool integrations are modular, hard-asserting declarations:** each integration "asserts hard on the structure produced by such apps so that they break quickly on changes and can be updated immediately" — brittleness here is intentional fail-fast against upstream drift, not a defect.
The quiverapp module's working shape: understand how quiverapp generates its tikz, open quiver in an iframe, inject a floating hover control that "exports" the tikz back into the app as a filter-compatible tikz diagram.
**The app never owns tikz generation** — it only renders SVGs via the filter layer ([Rendering Pipeline Requirements: Filters, MathJax, References](rendering-pipeline-requirements-filters-mathjax-references)).

**Original diagram-tool registry shape** (modular declaration per tool): `{id, executable, extension, format}` — e.g. qtikz/.tikz/TikZ, inkscape/.svg/SVG, ipe/.ipe/Ipe-XML. The extension+format columns are what the hard structure assertions key on.
Under the richness bar, each registry entry lives in that tool's own individually-managed plugin.

**Allowlist update (2026-06-13, supersedes the seed packet):** focus is quiverapp, qtikz, ipe (with FreeTikZ/Tikzit/Inkscape-class tools in spirit); **xournalpp is dropped** along with drawio — note the seed REQUIREMENTS.md still lists xournalpp as supported; the user's later statement wins and the contract docs need that correction.

**Figures: centralized and global-only.** Clipboard image → markdown image stored in the centralized figures directory (the vim predecessor attached figures to the project; "we do not want that going forward, since global images can simply be symlinked into any project if needed"). The directory is partially app-managed; the library must support **re-opening previously created images in their source tool** (as the vim setup did for Inkscape figures — `inkscape-figures.sh` returned the insertion snippet on create).
**Default location (user ruling, 2026-06-13): the sample config that ships with the app defaults the figures dir to `~/.pandoc/figures`, always.**

**Zotero:** integration should be extremely good and "likely owned by Zotero's own popup functionality" (CAYW — the original Tauri branch already integrated it via reqwest).
Hard gating is acceptable and preferred: **"the entire app can require a Better BibTeX Zotero export to even function."** A missing/misconfigured Zotero export is a fatal setup failure, not a degraded mode.
