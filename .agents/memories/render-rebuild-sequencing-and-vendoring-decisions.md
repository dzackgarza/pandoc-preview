# Render Rebuild: Sequencing and Vendoring Decisions

**When this applies:** planning or implementing the render-core rebuild, the plugin system, macro handling, or how `~/.pandoc` assets are installed. Three forks the user ratified 2026-06-14, closing open questions in [Feature Catalogue and Implementation Status](feature-catalogue-and-implementation-status), [MathJax Macro System: Tiers and Injection](mathjax-macro-system-tiers-and-injection), and [Required Filter Set](required-filter-set).

**Context:** greenfield2's `render.rs` still violates the ratified [Renderer Plugin Architecture](renderer-plugin-architecture) (renderer knowledge in the app core) and the [Pandoc Command Model and Raw String Contract](pandoc-command-model-and-raw-string-contract) (structured `pandoc.path`/`from_format`/`extra_args` in `config.toml`). All the requested math-document features (amsthm, tikz, citations, macros, the math insertion bar) are Tier-2 and sit on that foundation, so the foundation is rebuilt first.

**Fork 1 — Sequencing: FULL PLUGIN SYSTEM FIRST.** Build the generic plugin system (the Tier-4 "firewall") before any math feature is visible: TOML manifests `{id,name,description,category}` + a declarative (JSON-Schema-class) config schema + declared doctor checks; ONE generic validator the app runs over its own core schema AND every plugin's config section (zero plugin-specific knowledge in core); schema-driven config pages with a gum-driven custom-page hook; structured success/failure results (no `{ok?}`); plugin failure = toast; individually install/enable/disable/update/QC'd; plugins contribute doctor checks ([Plugins, Diagrams, Figures Requirements](plugins-diagrams-figures-requirements), [Shipped Config vs Runtime Defaults](shipped-config-vs-runtime-defaults)). Both renderers (pandoc + generic) ship as plugins from day one; acceptance test is that a `markdown-it` generic renderer works with no app changes. Rejected: the "interim raw-command swap on the current render.rs" (faster preview but rebuilds the seam later).

**Fork 2 — Macros: SPLIT BY PIPELINE.** HTML preview uses the existing `~/.pandoc` MathJax-injection pipeline (`generate-mathjax-config.py` → `mathjax-macros.json` + `math-macros.html` partial → injected into `pandoc_preview_template.html` between the `<!--MATHJAX_MACROS_START/END-->` markers; MathJax reads `window.MathJax.tex.macros` client-side). Export uses the user's macros in the LaTeX preamble via the md→tex→latexmk path (`dzg-unified` texmf). The app never embeds a macro list; it drives regeneration and re-renders when macros change. Accepted cost: tier3 TeX-only macros render in PDF only, diverging from preview — by design. Rejected: "de-macro everything via an intermediate LaTeX transform" (no turnkey solution, highest risk) and "MathJax-only".

**Fork 3 — Vendoring: SYMLINK FROM A VENDOR DIR.** The app keeps canonical copies of its shipped filters/templates/macro-toolchain in a bundled/XDG vendor dir and SYMLINKS them into `~/.pandoc/{filters,templates}` (canonical location per [Required Filter Set](required-filter-set)); updates are atomic, the app is the source of truth, and the user overrides any component by replacing its symlink with a real file. Doctor verifies the required symlink set is present/current and that external deps (`dzg-unified` texmf) exist — never vendor-copied. Rejected: "managed install + manifest with copy-and-confirm" and "copy once then fully user-owned".

**Open decisions still to settle in-plan (not blocking the plugin-system/renderer rebuild):** preview reader stays `-f markdown` (exact; supports `--citeproc` + raw-TeX; forgoes precise `sourcepos`/scroll-sync — defer `commonmark_x` until scroll-sync hardens, see [Decision Provenance: User-Owned vs Framework-Forced](decision-provenance-user-owned-vs-framework-forced)); citation-in-preview mechanism (`--citeproc`+CSL vs tex-path-only); figures dir canonical value (`~/.pandoc/figures` per [Shipped Config vs Runtime Defaults](shipped-config-vs-runtime-defaults) — confirm).
