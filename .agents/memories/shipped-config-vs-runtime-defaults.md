# Shipped Config vs Runtime Defaults

**When this applies:** any work on config loading, "defaults," the doctor/CLI, first-run, or deciding whether a value is allowed to have a default. User-ratified 2026-06-13. This refines the no-runtime-defaults rule with the single sanctioned use of a static defaults artifact.

**No constants.** Everything that would be a hardcoded constant is instead an explicitly configurable value that is simply validated — figures dir, debounce, font size, pandoc path, render command, the lot. No magic literal participates in a runtime decision ([First-Run Config Bootstrap Pattern](first-run-config-bootstrap-pattern)).

**What the app owns — and only this.** The app defines its own *core* config *schema* and *validates/enforces config at boundaries* (load, save, doctor). It validates plugin config the standard plugin-interface way: each plugin ships a declarative config schema in its manifest, and the app runs ONE generic validator over every plugin's config section against that plugin's own schema — so it validates plugin configs while holding zero plugin-specific knowledge ([Plugins, Diagrams, Figures Requirements](plugins-diagrams-figures-requirements)). That is the whole of its config responsibility. It does **not generate config** — not at runtime, and not via any "generate config" subcommand. There is no Rust config generator. Producing/authoring a config is offloaded to standard external tooling (gum), exactly as [Renderer Plugin Architecture](renderer-plugin-architecture)'s plugin-config and [First-Run Config Bootstrap Pattern](first-run-config-bootstrap-pattern)'s first-run flow describe. (This supersedes the first-iteration contract's "example-config generation is a separate explicit command" rule — user correction 2026-06-13.)

**Hard rule — no dynamic generation of config values, ever.** The app must never generate, guess, substitute, or fall back to a config value — not at runtime, not via a generator command. Missing/invalid config is a fatal, named error ([Threat Model: Polished Fallback Machine](threat-model-polished-fallback-machine)).

**The one sanctioned static-defaults artifact.** The app MAY ship a **static config file** — a hand-authored, version-controlled asset that *encodes* the default suggestions. It is a file checked in alongside the app, NOT something Rust emits. This is allowed precisely because it is a static asset, not a code path. **The app must almost never *use* the shipped config for any real runtime decision.** Its only legitimate uses are diagnostics and diffs:

- **Doctor/CLI diff:** the CLI reports how the user's config differs from the shipped defaults, so "what have I customized" is answerable at a glance.
- **Reset to defaults:** a CLI command copies the shipped config over the user's config to restore a known-good baseline, with confirmation (gum confirm guards the overwrite).

If runtime logic ever reads the shipped config to decide behavior, that is the fallback anti-pattern in disguise — fix it. The shipped config is a diagnostic baseline, never a value source.

**Gum modularizes would-be app-owned config UX.** Many concerns that would naively become app-owned config UI can be handed to standard external tools — gum especially. The first-run walkthrough, the reset-to-defaults confirm, and reconfiguration flows are gum scripts ([First-Run Config Bootstrap Pattern](first-run-config-bootstrap-pattern), [Doctor Contract (D1–D5)](doctor-contract)). This extends into plugins: **the entire pandoc flag-configuration setup can be gum-driven inside the pandoc renderer plugin** rather than a bespoke app GUI ([Renderer Plugin Architecture](renderer-plugin-architecture), [Plugins, Diagrams, Figures Requirements](plugins-diagrams-figures-requirements)). When scoping any config surface, first ask whether gum (or another standard tool) can own it before writing app code.
