# Preview Iframe and Asset Resolution

**When this applies:** implementing or modifying the preview pane, asset loading, or render-output injection. These are original-repo lessons (May 2026, Express + Tauri eras) that directly underpin the current P1/P5 obligations.

**Why the preview is an iframe:** pandoc `--standalone` output carries its own document-level CSS (e.g. `body { max-width: 36em }`) which, injected as `innerHTML`, bleeds into and constrains the whole app page (original commit 5f0a183: "restore iframe for preview"). The iframe confines the template's document semantics. Note the first greenfield iteration used same-document injection instead — but only because its harness lacked `frameLocator` ([Decision Provenance: User-Owned vs Framework-Forced](decision-provenance-user-owned-vs-framework-forced)); the current obligations (P1 "preview iframe document") restore the iframe model.

**The base-href trap:** a sandboxed `srcdoc` iframe's base URL defaults to `about:srcdoc`, so document-relative asset paths (images, figures) silently fail.

**RESOLVED 2026-06-14 — the preview EMBEDS resources; it does NOT resolve files.** The pandoc renderer (`pandoc-renderer/render.sh`) renders preview HTML with `--embed-resources`, so document images are inlined as `data:` URIs at render time (pandoc resolves them via `--resource-path` = the open file's dir). The webview therefore performs ZERO file/asset resolution — there is no asset-protocol `<base href>` chain to break on path/encoding quirks. This replaces the earlier "asset-protocol `<base href>` chain" mechanism, which proved fragile: relative `<img src>` + an `asset://` base href stopped resolving (preview images blank), and the chain's correctness depended on `convertFileSrc` output and path length. User ruling: the preview must not resolve files at all. **MathJax is the deliberate exception** — it stays an `asset://` `<script>` reference (pandoc cannot fetch `asset://`, so `--embed-resources` leaves the tag intact); the ~1MB engine loads once at runtime rather than being re-inlined on every keystroke. P5/`p05` still asserts the outcome (`img` decodes to 64×48) and is satisfied by the data URI — the spec is mechanism-agnostic. The `<base href>` is retained as an inert backstop only. **Never** reintroduce reliance on webview-side relative-asset resolution; **never** resurrect Express static routes or hand-rolled regex HTML rewriters that extract and rewrite src URLs.

**The keyboard-swallowing trap:** once the iframe has focus, global app shortcuts (quick-open, save, citation insert) die — keydown events stay inside the iframe document. Both original eras solved this by injecting a small forwarder into the preview HTML that re-posts keydown events to `window.parent` (main branch `src/client/components/PreviewPane.tsx:15-34`; tauri branch equivalent). Any preview implementation needs this or shortcuts break exactly when the user is reading their render.

**Update discipline:** push-based updates only (the 300 ms nvim poll loop was the app's first burned performance mistake); debounced render requests with latest-wins coalescing ([Reference: Render Lifecycle (Tideflow, mdTeX)](reference-render-lifecycle-tideflow-mdtex)); scroll position captured before injection and restored post-layout in `requestAnimationFrame` with a programmatic-scroll guard.

**Scroll sync** (full source↔preview design — pandoc `sourcepos` reader constraint, same-origin iframe access without postMessage, staged proportional→line-mapping plan) is scoped in [Reference: Source-Preview Scroll Sync](reference-source-preview-scroll-sync). Note the same-origin (`allow-same-origin`) iframe lets the parent read `contentDocument`/`scrollY` directly — relevant to both the keyboard forwarder and scroll listeners injected into the preview HTML.
