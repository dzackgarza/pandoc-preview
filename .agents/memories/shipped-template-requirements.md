# Shipped Template Requirements

**When this applies:** authoring, modifying, validating, or shipping any template the app depends on. The live-preview template is `~/.pandoc/templates/pandoc_preview_template.html` (51KB, verified 2026-06-13 — it carries the MathJax markers at L27–39 and the hover-edit/postMessage JS at L174–224, and the ppe tauri branch's own `config.toml` wires exactly this template into its `render_command`). Governing rule: **templates are data, not code** — user-editable files in `~/.pandoc/templates/`, never embedded in app source, never constructed by string manipulation, existence-validated at startup, referenced only from the render command.

**The five MUST properties below govern the live-PREVIEW template** (the one the pandoc renderer pipes through for the editor's preview pane). `pandoc_preview_template.html` is the model and already satisfies all five. A shipped preview template MUST:

1.  **Carry the full MathJax 3 configuration with user macros injected** between the `<!--MATHJAX_MACROS_START/END-->` markers, regenerated via the `~/.pandoc` injection pipeline ([MathJax Macro System: Tiers and Injection](mathjax-macro-system-tiers-and-injection)) — macro lists are generated, never hand-edited; the deep-MathJax requirement is satisfied BY the template.
2.  **Include the modular hover-edit JS** implementing the postMessage contract (verified at template lines ~151-250): query `.pandoc-preview-editable, img, embed`; hover shows an Edit overlay; click sends `window.parent.postMessage({type: 'pandoc-preview-edit', kind, path}, '*')` with `kind ∈ {tikzcd, tikzpic, tikzcode}` (no path) or image kinds (path from src). Template-side JS + filter-provided hooks is the architecture; the app only listens — it never scrapes rendered HTML.
3.  **Provide the visual layer for filter-emitted semantics:** `.pandoc-preview-editable` hover affordance CSS, theorem/proofenv environment styling, and callout styling (the env/callout CSS lives in the `templates/css/` layer, e.g. `math-environments.css` — a template must pull in what its filters' output needs).
4.  **Be `--standalone`-compatible and iframe-confined:** its document-level CSS (body widths, fonts) is expected and is exactly why the preview is an iframe ([Preview Iframe and Asset Resolution](preview-iframe-and-asset-resolution)); the preview must render correctly with NO app-side CSS leaking in either direction.
5.  **Resolve assets through the current chain** — absolute refs / Tauri asset-protocol (P5's decoded-pixels proof) — with no Express-era `/api/preview-assets/` or host-header base-href assumptions.

**Vendored templates installable into `~/.pandoc/templates/` (user requirement, 2026-06-13):** the app VENDORS the user's templates and INSTALLS them — the same install surface as the pandoc plugin's filters ([Required Filter Set](required-filter-set)). Verified inventory:

- **`pandoc_preview_template.html`** — the live-preview template (above); the pandoc renderer plugin ships and manages it.
- **`research_draft.html`** — a research-document OUTPUT template, NOT the preview template. Small (1.9KB), partial-based: pulls `${css/math-environments.css()}` into an inline `<style>` and injects math macros via the `\({${css/math-macros.html()}}\)` inline-math trick. It is **exempt from the five preview-MUST properties** — those govern the preview pane, and this template is a document-output artifact the user renders deliberately. Its only install requirement: carry its partials from `templates/css/` (`math-environments.css`, `math-macros.html` — the latter generated by the MathJax macro pipeline alongside `mathjax-macros.{json,mjs,ts}`). (My earlier note treating it as a preview template that "must be reconciled against the five MUST properties" was wrong — corrected here.)
- **`research_draft.tex`** — the "real paper" PDF template (the user's research_draft PDF template): `amsart` + the user's `dzg-unified` texmf package (provides amsmath/amsthm/mathtools/tikz/tikz-cd/…), `\ifshort` toggle, hyperref with backref. `dzg-unified` lives in texmf, NOT `~/.pandoc` — an external hard dependency the installer/doctor must check, never vendor-copy. (`research_paper.tex` also exists; only the two `research_draft.*` plus the preview template are ratified as vendored so far.)

The **tikz wrapper template** (`standalone-tikz.tex`) has its own contract: a fixed literal file on disk containing the required packages plus the user's macro/preamble subset (so `\RR`, `\coxeterblacknode` work inside diagrams), with exactly one content slot — no dynamic template construction in Lua/JS, ever (the "Fixed TeX Compiling" design constraint).

**Non-shipped templates are first-class users:** the user may point the render command at any template (revealjs, impress, etc. all exist in `~/.pandoc/templates/`) and the app renders whatever HTML results — a shipped template earns no special treatment in app code; "shipped" only means the app guarantees the relevant properties for the templates it installs and manages.

**Verify:** render the witness fixture through `pandoc_preview_template.html` — math typesets with user macros (P4), theorem/callout blocks are styled, hovering a tikz block shows the edit affordance and clicking posts the message, images decode through the asset chain (P5). For vendored research templates: a fresh install places template + partials into `~/.pandoc/templates/` and a render through `research_draft.html` typesets macros from the generated partial.
